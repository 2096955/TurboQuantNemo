   Path: scripts/replay_npt16_capture.py against the .npz produced by
   ISOQUANT_CAPTURE_PATH instrumentation.
   Result: kernel vs _fused_attention_mlx linf=9.31e-09, cosine=1.000000.
   The kernel is bit-equivalent to its own fallback at fp32 noise on
   real inputs. NO SINGLE-CALL KERNEL CORRECTNESS BUG.

C. Synthetic distribution sweep. Five variants (Gaussian tame pe,
   sharp pe N(0,5), pe with finfo.min sentinels, Student-t df=3 K,V,
   combinations). All variants kernel-vs-fb linf at 1e-6 noise.
   Synthetic distributions don't reproduce the trajectory gap.

D. Algorithmic-equivalence analysis (cross-combine arithmetic test).
   Computed four outputs with explicit ops on captured tensors:
     O_rr = invR(softmax(q_rot @ k_rot.T + m) @ v_rot)
     O_oo = softmax(q_orig @ k_orig.T + m) @ v_orig
     O_ro = softmax(q_rot @ k_rot.T + m) @ v_orig
     O_or = invR(softmax(q_orig @ k_orig.T + m) @ v_rot)
   All four agree pairwise to 1.5e-08. So rotated-vs-original-space
   attention is mathematically equivalent under explicit fp32 matmul.

E. Forced same-codebase A/B (Codex's recommended discriminator).
   Added ISOQUANT_FORCE_RECONSTRUCT_ATTENTION=1 flag to skip the fused
   path and route through reconstruct+mx.fast.SDPA on identical
   post-fix code. Result: top1=0.9091 (with iso_path_observed=
   unfused_reconstruct, unfused_fallback_calls=3304, decompress_calls=
   9912). Confirms the 11pp gap is real on current codebase, not a
   historical artifact.

F. Precision-regime mechanism investigation. Found that:
     reconstruct_keys() returns FP16  (cast at mlx_isoquant.py:1133)
     get_values() returns FP16        (cast at mlx_isoquant.py:1150)
     _decompress_batch() returns FP32 (used internally by other paths)
   The reconstruct path's mx.fast.SDPA receives fp16 K/V; the kernel
   and _fused_attention_mlx operate fp32 throughout. The reference path
   (default cache) also stores fp16 K/V. So reference + reconstruct
   share BOTH the SDPA implementation AND the fp16 precision regime;
   kernel/_fused_attention_mlx use fp32 internally and drift from the
   fp16 reference's precision noise pattern.

   Confirmation test: cast K, V to fp16 before rotated-space matmul.
   Result: did NOT bring rotated-space output closer to reconstruct
   (linf 1.16e-05 vs 9.29e-06 — slightly farther). Precision-regime
   contributes (A vs B = 7e-6 confirms fp32-vs-fp16 sensitivity exists)
   but doesn't fully explain the rotated-vs-original gap.

G. State-asymmetry hypothesis (Codex hypothesis B). Tested: is
   _packed_latent_cache (kernel reads) ever out of sync with
   pack_indices_3bit(_compressed_latent["indices"]) (fb-equivalent reads)?
   Drove a synthetic cache through prefill+finalize+5 decode steps,
   asserted parity at every checkpoint. Max byte diff = 0 throughout.
   State asymmetry refuted.

H. metric-bug discovery. Earlier 4.3e-02 linf measurements (which had
   driven much of mid-session reasoning) turned out to be a numpy
   broadcast artifact: out_kernel (64,512) - out_recon (1,64,1,512)
   broadcast to (1,64,64,512) phantom comparison over 2M elements
   instead of the actual 32K outputs. After reshape fix, real linf is
   9.28e-06. Important enough to warrant the entire diagnostic
   reorientation around real-magnitude differences.

PITFALLS TO AVOID (lessons from this session)
-----------------------------------------------
  - Don't trust setup_seconds_iso < setup_seconds_ref as proof the fused path
    ran. Always check `iso_path_observed` in the artifact (added in 423abe0).
    Older artifacts pre-dating that field can't tell you which path ran.
  - Don't run two back-to-back full validations without checking memory
    pressure — expert offload + 554 GB model leaves little headroom.
  - The fused gate is bit_width == 3 specifically. --bits 4 falls into
    the reconstruct path automatically; that's the intended A/B, not a bug.
  - ISOQUANT_USE_NPT16_FUSED defaults to 0 in the env. To exercise the
    fastest fused path you MUST set it to 1 explicitly.
  - The iso run has internal Metal timeouts that can leave the process in
    UE state for 40+ minutes. If you see UE in `ps`, do NOT issue another
    iso run on the same GPU — it will compound. Reboot first.
  - Don't add features to the harness while the validation is unproven.
    Codex item #2 (move sync into finalize_deferred_prefill) is intentionally
    deferred until bits=3 per-step lands so we don't conflate causes.
